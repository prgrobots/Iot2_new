    Session 05: I²C & Actuators¶

    Week: 5
    Element: ICTIOT503 Element 3: Produce design documentation | ICTIOT502 Element 2: Program IoT device
    Duration: 3.5 hours
    Phase: Electronics for Programmers

    Note: All electronics sessions can be completed using either Arduino (C++) or MicroPython on the ESP32. You may choose your preferred language for all programming tasks, code submissions, and portfolio checkpoints.

    Second MCU — Raspberry Pi Pico W: This session introduces a second microcontroller into the course. The Raspberry Pi Pico W used here has an OLED display and MPU6050 IMU soldered directly to its carrier board, providing a reliable I²C connection ideal for learning. Pico W tasks in this session use MicroPython.

    Session Introduction¶

    This week you'll add the OLED display and accelerometer (GY-521/MPU6050) to your truck, plus a servo motor for simulating the dump bed. You'll learn I²C bus communication — a critical protocol for industrial sensor networks — on two different microcontrollers: the ESP32 (using C++ or MicroPython) and the Raspberry Pi Pico W (MicroPython only). Working across both platforms is a key mapping requirement for this unit of competency.

    This session launches Assessment 3 (Tire & Suspension Health Monitor), which will be completed on the Raspberry Pi Pico W, and Assessment 4 (Payload & Load Management), which will be completed on a separate MCU using header pins on a solderless breadboard.

    Learning Objectives¶

    By the end of this session, you will be able to:

    • Configure and use I²C communication protocol
    • Display sensor data on OLED screen
    • Read accelerometer data from MPU6050 (GY-521)
    • Control servo motors with precise positioning
    • Read a microcontroller pinout diagram to identify I²C and power pins
    • Use GPIO output pins to power I²C devices on the Raspberry Pi Pico W
    • Explain why solderless breadboards can cause unreliable I²C connections
    • Start Assessments 3 & 4 design documentation

    Session Structure¶

    1. I²C Protocol – Theory and wiring
    2. OLED Display (ESP32) – Real-time data visualisation
    3. MPU6050 Accelerometer (ESP32) – Vibration and tilt detection
    4. Servo Control (ESP32) – Dump bed simulation
    5. LDR Dimming – Analog light sensing to control LED brightness
    6. Raspberry Pi Pico W Track – Pinout, GPIO power control, I²C with soldered devices
    7. Assessment Launch – Design documentation requirements

    RockCore Mining Context¶

    Modern haul trucks use I²C sensor networks to monitor dozens of data points simultaneously. Today you'll implement:

    • OLED Display (Base panel): Shows truck status at a glance
    • GY-521 Accelerometer (Base panel): Detects tire imbalance and suspension issues
    • Servo Motor (Back panel): Simulates hydraulic dump bed for payload management

    I²C allows all these devices to share just two wires (SDA/SCL) plus power – essential when space is limited.

    Task 2: GY-521 Accelerometer (Vibration Detection)¶

    Wiring: Shares same I²C bus as OLED (daisy-chain SDA/SCL).

    // Week 5 Task 2 - GY-521 Accelerometer
#include <Wire.h>
#include <MPU6050.h>

MPU6050 mpu;

void setup() {
  Serial.begin(115200);
  Wire.begin();

  if (!mpu.begin(MPU6050_SCALE_2000DPS, MPU6050_RANGE_2G)) {
    Serial.println("MPU6050 not found - check wiring");
    while(1);
  }
}

void loop() {
  Vector normAccel = mpu.readNormalizeAccel();

  Serial.print("X: ");
  Serial.print(normAccel.XAxis);
  Serial.print(" | Y: ");
  Serial.print(normAccel.YAxis);
  Serial.print(" | Z: ");
  Serial.println(normAccel.ZAxis);

  // Vibration detection (for Assessment 3)
  float totalAccel = sqrt(normAccel.XAxis * normAccel.XAxis + 
                          normAccel.YAxis * normAccel.YAxis + 
                          normAccel.ZAxis * normAccel.ZAxis);

  if (totalAccel > 1.5) {  // Threshold tuning required
    Serial.println("WARNING: High vibration - tire imbalance?");
  }

  delay(100);
}

    Task 3: Servo Motor Control (Dump Bed)¶

    Wiring: - Servo signal → GPIO 18 - Servo VCC → 5V (external power recommended for full-size servos) - Servo GND → GND

    // Week 5 Task 3 - Servo Dump Bed Simulation
#include <ESP32Servo.h>

Servo dumpServo;
const int SERVO_PIN = 18;

void setup() {
  Serial.begin(115200);
  dumpServo.attach(SERVO_PIN);
  dumpServo.write(0);  // Start position: bed down
  Serial.println("Dump bed ready");
}

void loop() {
  // Simulate dump cycle
  Serial.println("Raising dump bed...");
  for(int pos = 0; pos <= 120; pos++) {
    dumpServo.write(pos);
    delay(15);
  }
  delay(2000);

  Serial.println("Lowering dump bed...");
  for(int pos = 120; pos >= 0; pos--) {
    dumpServo.write(pos);
    delay(15);
  }
  delay(5000);
}

    Raspberry Pi Pico W Track: I²C with Soldered Devices¶

    Why a Second MCU?

    The Raspberry Pi Pico W is introduced here because Assessment 3 (Tire & Suspension Health Monitor) must be completed on this board. The Pico W carrier board used in this course has the OLED display and MPU6050 IMU soldered directly to the PCB — no breadboard jumpers needed for I²C connections. This models industrial practice where permanent, soldered connections are essential for signal reliability.

    Assessment 4 (Payload & Load Management) will be completed on a different MCU using a solderless breadboard with header pins. That contrast — reliable soldered I²C vs. breadboard I²C — is a key learning outcome for this unit of competency.

    Pico W Hardware Overview¶

    Device Interface I²C Address Connection Type
    OLED (128×64 SSD1306) I²C 0x3C Soldered to carrier board
    MPU6050 IMU I²C 0x68 Soldered to carrier board

    Both devices share a single I²C bus. Because they are soldered, the connections are permanent and low-resistance — meaning no intermittent "device not found" errors from a loose jumper wire.

    Step 1: Read the Pico W Pinout¶

    Critical First Step

    Before writing a single line of code, locate and download the official Raspberry Pi Pico W Pinout PDF. You need to identify:

    • Which GPIO pins are the I²C SDA and SCL lines (default: GP4 = SDA, GP5 = SCL on I2C0)
    • Which GPIO pins are used to supply power (VCC) to the OLED and MPU6050 on your specific carrier board

    On this carrier board, the OLED and MPU6050 do not draw power directly from the 3.3V rail. Their VCC lines are wired to GPIO output pins — meaning you must set those pins HIGH in your code before the devices will power up and respond on the I²C bus.

    Check Your Board

    Look at your carrier board or ask your instructor for the schematic. Note down the pin numbers for:

    • OLED VCC pin → GP___
    • MPU6050 VCC pin → GP___

    Write them in your portfolio. These are real-world skills — reading a schematic or pinout to understand how hardware is wired is a core competency in this unit.

    Step 2: Power the I²C Devices via GPIO¶

    # Pico W - Step 2: Bring VCC pins HIGH to power I2C devices
# Replace VCC_OLED_PIN and VCC_IMU_PIN with the actual GPIO numbers
# from your carrier board pinout/schematic.

from machine import Pin
import time

VCC_OLED_PIN = 15   # CHECK YOUR BOARD — replace with correct pin
VCC_IMU_PIN  = 16   # CHECK YOUR BOARD — replace with correct pin

oled_pwr = Pin(VCC_OLED_PIN, Pin.OUT)
imu_pwr  = Pin(VCC_IMU_PIN,  Pin.OUT)

oled_pwr.value(1)   # HIGH = VCC on → OLED powered
imu_pwr.value(1)    # HIGH = VCC on → MPU6050 powered

time.sleep_ms(100)  # Allow devices to settle
print("I2C devices powered on")

    What Happens If You Skip This?

    If you forget to bring the VCC pins HIGH, the OLED and MPU6050 have no power. When you scan the I²C bus, you will see:

    Found I2C devices: []

    No devices at any address. This is exactly the same error you would see with a bad breadboard connection — and that is the teaching point. Whether it is a loose breadboard jumper or a GPIO pin left LOW, the result is identical: the device is unreachable on the bus.

    Pico W Task 1: I²C Scanner¶

    Always run an I²C scanner first to confirm both devices are found before writing application code. This is good practice and saves debugging time.

    # Pico W Task 1 - I2C Scanner
from machine import Pin, I2C
import time

# Power on I2C devices (update pin numbers from your board schematic)
VCC_OLED_PIN = 15
VCC_IMU_PIN  = 16
Pin(VCC_OLED_PIN, Pin.OUT).value(1)
Pin(VCC_IMU_PIN,  Pin.OUT).value(1)
time.sleep_ms(100)

# Initialise I2C bus (I2C0: SDA=GP4, SCL=GP5)
i2c = I2C(0, sda=Pin(4), scl=Pin(5), freq=400_000)

# Scan for devices
devices = i2c.scan()

if devices:
    print("Found I2C devices:")
    for addr in devices:
        print(f"  Address: {hex(addr)}")
else:
    print("No I2C devices found — check power pins and wiring!")

    Expected output: 

    Found I2C devices:
  Address: 0x3c   ← OLED
  Address: 0x68   ← MPU6050

    If you see an empty list, revisit Step 2 and confirm your VCC pin numbers.

    Pico W Task 3: MPU6050 Accelerometer (MicroPython)¶

    # Pico W Task 3 - MPU6050 Accelerometer
# Uses raw register reads — no external library needed

from machine import Pin, I2C
import struct
import time

# Power on devices
Pin(15, Pin.OUT).value(1)   # Update pin number for your board
Pin(16, Pin.OUT).value(1)
time.sleep_ms(100)

i2c = I2C(0, sda=Pin(4), scl=Pin(5), freq=400_000)

MPU_ADDR = 0x68

# Wake the MPU6050 (it starts in sleep mode)
i2c.writeto_mem(MPU_ADDR, 0x6B, b'\x00')
time.sleep_ms(100)

def read_accel():
    raw = i2c.readfrom_mem(MPU_ADDR, 0x3B, 6)
    ax, ay, az = struct.unpack(">hhh", raw)
    # Convert to g (±2g range, sensitivity = 16384 LSB/g)
    return ax / 16384.0, ay / 16384.0, az / 16384.0

while True:
    x, y, z = read_accel()
    print(f"X: {x:.3f}g  Y: {y:.3f}g  Z: {z:.3f}g")

    total = (x**2 + y**2 + z**2) ** 0.5
    if total > 1.5:
        print("WARNING: High vibration — tire imbalance?")

    time.sleep_ms(200)

    I²C Reliability: Solderless Breadboard vs Soldered Connections¶

    Teaching Point — Why Devices 'Disappear'

    One of the most common I²C debugging problems is a device that was working and then stops responding — or never responds at all. The error is always the same: no device found at that address.

    Common causes on a solderless breadboard:

    Cause Effect
    Loose jumper wire in breadboard hole Intermittent open circuit on SDA or SCL
    Two wires sharing a row incorrectly Short circuit between SDA and SCL
    Breadboard strip damaged or corroded High contact resistance, signal corruption
    VCC jumper not fully seated Device unpowered, no I²C response

    On the Pico W carrier board, the OLED and MPU6050 are soldered — every joint is permanent and low-resistance. You will not see these intermittent failures.

    For Assessment 4, you will wire an MCU to sensors on a solderless breadboard. Pay close attention to connection quality. If a device disappears from the I²C bus mid-session, check your breadboard connections first.

    Debugging Workflow

    1. Run the I²C scanner — confirm all expected addresses appear
    2. If an address is missing: check power (VCC/GND), then SDA, then SCL
    3. Try a different breadboard row or a different jumper wire
    4. Reduce I²C frequency to 100_000 Hz if signals are marginal

    LDR Dimming — Ambient Light to LED Brightness¶

    A photoresistor (LDR) outputs a varying analog voltage depending on light level. We can map this directly to the brightness of the RGB LED from Session 4 using PWM — the brighter the ambient light, the dimmer the status LED (simulating auto-brightness in the truck cabin).

    Session 6/7

    This LDR task is covered here in session ⅚. It will not appear in session 6/7 — consider it done once you complete the step below.

    Wiring:

    3.3V ──── [LDR] ──── GPIO 35 ──── [10 kΩ resistor] ──── GND

    Connect the RGB LED red channel to GPIO 32 (or the pin used in session 4).

    // Week 5/6 — LDR dimming the RGB LED (red channel)
const int LDR_PIN = 35;
const int LED_PIN = 32;

void setup() {
  Serial.begin(115200);
  ledcSetup(0, 5000, 8);       // PWM channel 0, 5 kHz, 8-bit resolution
  ledcAttachPin(LED_PIN, 0);
}

void loop() {
  int ldrValue = analogRead(LDR_PIN);   // 0 (dark) to 4095 (bright)

  // Invert: bright room → dim LED; dark room → bright LED (like auto-brightness)
  int brightness = map(ldrValue, 0, 4095, 255, 10);
  ledcWrite(0, brightness);

  Serial.printf("LDR: %d → LED brightness: %d\n", ldrValue, brightness);
  delay(200);
}

    Cover the LDR with your finger (dark) and watch the LED brighten; uncover it (bright) and it dims.

    Check Your Knowledge¶

    Q1 – I²C Addressing

    How can multiple I²C devices share the same two wires without conflicts?

    Answer

    Each I²C device has a unique address (e.g., OLED=0x3C, MPU6050=0x68). The master (ESP32 or Pico W) specifies which address to communicate with, and only that device responds.

    Q2 – Accelerometer Applications

    Besides vibration, what else can the MPU6050 detect for your truck?

    Answer

    Tilt angle (is truck on steep incline?), sudden impacts (collision detection), and orientation (is truck tipped over?).

    Q3 – Servo Power

    Why might you need external power for the servo instead of ESP32's 5V pin?

    Answer

    Servos can draw 500mA+ under load. ESP32's 5V pin is limited to ~500mA total, which could brown out the microcontroller when the servo moves.

    Q4 – GPIO Power Control on Pico W

    On the Pico W carrier board, why are the OLED and MPU6050 VCC lines connected to GPIO output pins rather than the 3.3V rail?

    Answer

    Connecting VCC to a GPIO output pin lets the microcontroller power-cycle each device individually — useful for fault recovery and power management. It also means a device will not respond on the I²C bus if the GPIO pin is forgotten or left LOW, which is a deliberate learning exercise in this session.

    Q5 – Breadboard Reliability

    You're testing your Assessment 4 circuit and the MPU6050 was responding on address 0x68, but after moving the breadboard it no longer appears in the I²C scan. What is the most likely cause and how do you fix it?

    Answer

    A jumper wire has come loose from its breadboard hole, breaking the SDA or SCL (or VCC) connection. Re-seat all jumper wires firmly, especially those connected to the sensor. Run the I²C scanner again to confirm the device reappears.
